Correlation

Every log line carries a trace_id and span_id so you can trace one request across every service end-to-end.

Resilience

Kafka decouples producers from consumers. Services never block on logging. If your observability stack goes down, logs queue safely.

Separation

Audit logs go to a separate, append-only topic and WORM store — completely isolated from debug logs and untouchable by service code.

Standard Log Schema (JSON)

All services emit structured JSON. No free-text log messages. Fields are mandatory unless marked optional.

Severity Levels
ERROR WARN INFO DEBUG AUDIT
ERROR — Requires immediate attention / alerts
WARN — Degraded state, needs investigation
INFO — Normal business events (production)
DEBUG — Verbose detail (disabled in prod)
AUDIT — Security events (always on, immutable)
PII & Privacy Rules
  • ✦ Never log passwords, tokens, or secrets
  • ✦ Hash user IDs before logging (SHA-256 + salt)
  • ✦ Truncate card numbers: last 4 digits only
  • ✦ Mask emails: j***@example.com
  • ✦ Strip full IP addresses from app logs
  • ✦ Audit logs retain full identity (separate store)
Ingestion Pipeline
01

Emit to stdout/stderr

Services log structured JSON only. No log files. Container runtime captures output. Use pino, logrus, or structlog.

02

Fluent Bit Collector

Deployed as Kubernetes DaemonSet. Tails container logs, parses JSON, enriches with pod/namespace metadata. Buffers locally on node.

03

Kafka Topics

Separate topics: logs.app, logs.audit, metrics.perf. Partitioned by service name. Retention: 7 days on bus.

04

Stream Processing

Kafka Streams / Flink for anomaly detection, sampling, and routing. Drop DEBUG logs in prod. Flag ERRORs for immediate alerting.

05

Write to Stores

Hot logs → Elasticsearch. Metrics → Prometheus. Audit → WORM. Cold archive → S3 with Parquet partitioning by service+date.

Sampling Strategy
  • 📌 AUDIT / ERROR — 100% — never sample
  • 📌 WARN — 100% in prod, 50% staging
  • 📌 INFO — 100% prod, 10% high-volume paths
  • 📌 DEBUG — OFF in prod, 100% dev/staging
  • 📌 Traces — Head-based 10%, tail-based 100% for errors

Index Strategy (Elasticsearch)
  • ✦ One index per service per day: payment-2025.11.14
  • ✦ ILM policy: hot→warm at 30d, delete at 365d
  • ✦ Shard size: target 30–50GB per shard
  • ✦ Audit index: separate cluster, no delete policy
  • ✦ Force merge on rollover to reduce segment count
Cost Controls
  • ✦ Drop DEBUG logs before Kafka (save ~40% volume)
  • ✦ Compress with LZ4 in transit, Zstd at rest
  • ✦ Deduplicate repeated ERROR bursts (token bucket)
  • ✦ Sample high-frequency INFO events (>1k/s)
  • ✦ S3 Intelligent Tiering for cold archive
  • ✦ Review top-volume services monthly
